# ADR-002: Choosing Stripe Checkout Redirect Mode Over Stripe Elements

- **Status**: Accepted
- **Date**: 2026-07-17

---

## Context

To charge users for subscriptions, our SaaS prototype must interface with Stripe. We have to collect payment details (credit cards, billing address) and authorize recurring charges.

Stripe offers two primary development paths for card collection:
1. **Option 1: Stripe Elements**: Embedded UI components rendered directly inside our Next.js dashboard pages.
2. **Option 2: Stripe Checkout**: Cryptographically signed redirect sessions to pre-built billing forms hosted by Stripe.

---

## Decision

We chose **Option 2: Stripe Checkout Redirect Mode**.

Upon clicking "Subscribe", the Next.js server fires a secure request to Stripe to generate a Checkout Session, then returns a checkout URL. The user is redirected to Stripe's payment portal and returns to the dashboard after completing the transaction.

---

## Consequences

### Positive
- **Minimized PCI Compliance**: We never touch or render card inputs in our own HTML/DOM tree. Stripe handles all security and PCI compliance responsibilities.
- **Out-of-the-Box Features**: Stripe Checkout handles international tax collection, Apple Pay/Google Pay redirects, and local payment methods automatically without updating application code.
- **Zero Frontend UI Development**: Designing high-quality, validation-compliant payment forms is time-consuming. Offloading this to Stripe lets us focus on core product features.
- **Mobile Optimizations**: Stripe Checkout templates are fully responsive and optimized for mobile devices.

### Negative
- **Friction in User Experience**: Users must leave our custom application domain to make payments and wait to be redirected back, creating a slight pause in the user onboarding funnel.
- **Custom Branding Restrictions**: Customizing the payment portal is limited to branding adjustments (colors, logos) configurable in the Stripe Dashboard, rather than full custom HTML/CSS controls.
