# ADR-001: Selection of Supabase SSR for Authentication

- **Status**: Accepted
- **Date**: 2026-07-17

---

## Context

We need an authentication and session-tracking solution for a Next.js 14 App Router SaaS product. The authentication system must:
1. Provide secure signup, login, and logout capabilities.
2. Support Server-Side Rendering (SSR) to allow page-protection checks on the server before HTML rendering, avoiding client-side layout shifts.
3. Integrate with the database layer without introducing complex synchronizations or extra databases.

We considered two primary approaches:
- **Option 1**: NextAuth.js (Auth.js) combined with a database adapter.
- **Option 2**: Supabase SSR (`@supabase/ssr`) storing JWT session tokens in browser cookies.

---

## Decision

We chose **Option 2: Supabase SSR** (`@supabase/ssr`).

By using Supabase SSR:
- Authentication state and user tables reside inside the same ecosystem as our data layer (`auth` schema vs `public` schema in PostgreSQL).
- Web request cookies are synced automatically by `@supabase/ssr` between the server (middleware, server components, API handlers) and browser client.
- It eliminates the need to run and maintain a separate OAuth/credentials session adapter database layer.

---

## Consequences

### Positive
- **Visual Performance**: Next.js Server Components and Middleware read the auth state directly from cookie headers, preventing "flash of unauthenticated state" layout shifts.
- **Security**: Supabase database tables can query the secure session user ID directly in PostgreSQL using `auth.uid()`, enforcing Row Level Security (RLS) rules at the database level.
- **Reduced Complexity**: One less dependency to keep updated. Authentication and database queries use the unified Supabase JS library.

### Negative
- **Provider Vendor Lock-In**: Database authentication is tied to Supabase Auth. Switching from Supabase to another database provider in the future would require migrating users or re-architecting the session model.
- **SMTP Setup**: Supabase default signup emails are heavily rate-limited in production. We must configure custom SMTP provider settings (e.g., SendGrid, Resend) before launch.
